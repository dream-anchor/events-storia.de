### Findings

**P0 — LexOffice-Anzahlungsrechnung fehlt bei 3 bezahlten Deposits (compliance-relevant)**
3 bezahlte `deposit`-Zahlungen (`bd8d5d74…`, `897c9308…`, `52b72c98…`) haben
`lexoffice_invoice_id = NULL`. Ursache: der Multi-Option-Zweig in
`handle-stripe-webhook` (Zeile 830) triggert bei Anzahlungen
`create-manual-invoice`, aber **nicht** `create-lexoffice-downpayment-invoice`
wie im Maestro-Zweig (Zeile 663). Damit entsteht keine UStG-konforme
Anzahlungsrechnung — nur eine reguläre Rechnung oder gar keine.
→ Fix: Multi-Option-Pfad muss `triggerInvoiceForPayment(paymentId, inquiryId, paymentType)`
aufrufen; Backfill für die 3 Altfälle via manuelle LexOffice-Anlage.

**P1 — Idempotenz-Lücke in `create-balance-payment-link`**
Beim wiederholten Aufruf (z. B. „Link neu senden") wird jedesmal ein neuer
`v2_payments`-Eintrag mit `status='sent', payment_type='balance'` angelegt
(Zeile 152), obwohl der `balance_payment_links`-Datensatz via Slug korrekt
geupsertet wird. Timeline zeigt dann Duplikate. → Dedup per
`(event_id, payment_type='balance', status='sent')` vor Insert.

**P1 — Stripe-`apiVersion`-Mismatch zwischen Funktionen**
`create-offer-payment-link` verwendet `'2025-08-27.basil'`, `handle-stripe-webhook` und
`create-event-payment-session` `'2024-06-20'`. Kein Runtime-Bruch, aber unterschiedliche
Object-Shapes für PaymentIntent-Erweiterungen. → Auf `'2025-08-27.basil'` vereinheitlichen.

**P1 — Legacy-Schemata weiterhin im Webhook aktiv**
`handle-stripe-webhook` schreibt in `event_inquiries` (Zeile 722), `event_bookings`
(Z. 558), `catering_orders` (Z. 166) und `event_payments` (Z. 628) — parallel zu den
neuen `v2_*`-Tabellen. Solange die Legacy-Views noch existieren, ist das ungefährlich;
bei einer künftigen Legacy-Bereinigung müssen diese Pfade migriert oder entfernt
werden. Wichtig: der Maestro-Zweig aktualisiert **nur** `event_payments`, nicht
`v2_payments` — Zahlungen aus `create-event-payment-session` erscheinen nicht in der
v2-Statistik.

**P2 — 6 Orphan-Payments mit `evt.status='cancelled'`, aber `v2_payments.status='paid'`**
8 Fälle, davon 6 mit Kleinbeträgen (~18–22 €) auf 3 unterschiedlichen Events. Sieht nach
Test-/Refund-Szenarien aus (keine `is_test=true`-Flag, aber Beträge und
Wiederholung sprechen dafür). Nicht sicherheits-, aber reporting-relevant.
→ Prüfen ob diese Payments storniert / rückerstattet werden sollen.

**P2 — Aged `sent` / `overdue` Zahlungen ohne Follow-up**
- `fd353ccf…` — final 4.410 €, `sent` seit 29.05. (33 Tage) — Event `316a0f27…`
- `9e5513ec…` — prepayment 1.380 €, `overdue` seit 08.04. (85 Tage) — Event `8ca4230e…`
- `96edee70…` — deposit 500 €, `overdue` seit 08.04. (85 Tage) — Event `8ca4230e…`

Kein automatisches Mahnungs-/Expiry-Handling erkennbar (`reconcile-payment-statuses`
setzt nur `paid`-Sync, keine Aged-Detection). → Cron-Rule „nach X Tagen `overdue`,
Alert an Staff" ergänzen.

**P2 — `handle-stripe-webhook` gibt bei Fehlern immer `200` zurück (Zeile 127)**
Fehler werden geloggt und `reportEdgeError` aufgerufen, aber Stripe erfährt nichts.
Dadurch kein Retry — auch bei transienten DB-Ausfällen. Bewusstes Design (verhindert
endlose Retries), aber bei echten Ausfällen fehlt die Selbstheilung.
→ Bei 5xx-typischen Fehlern (DB-Timeout, LexOffice-503) sollte `500` zurückgegeben
werden, um Retry zu erlauben.

**P3 — `create-event-payment-session` ohne Auth-Prüfung**
Function erwartet nur `payment_id` und lädt darüber Kunden-/Betragsdaten. Kein
Admin-Role-Check wie in `create-offer-payment-link`. Zugriff via `verify_jwt` im Router
prüfen; Sitzungserstellung sollte auf Staff-Kontext beschränkt sein.

**Positiv**
- Signature-Verify (`constructEventAsync`) korrekt implementiert.
- Idempotenz-Checks in allen `handle*Payment`-Funktionen vorhanden.
- `reconcile-payment-statuses` als Sicherheitsnetz aktiv (48 h-Fenster).
- 23-h-Expiry für Stripe-Sessions korrekt (unter 24 h-Limit).

**Handlungsempfehlung (Reihenfolge)**
1. P0 UStG-Anzahlungsrechnung im Multi-Option-Zweig fixen + Backfill.
2. P1 Balance-Link Dedup + apiVersion vereinheitlichen.

## Ebene 2 P0-1 — Anfrage-Flow (E2E, 2026-07-01)

**Status: ✅ funktioniert** — mit 3 Findings

**Repro:** `/tmp/browser/p0-1/e2e.py` — füllt `/events/#kontaktformular`,
submittet, prüft Response + Success-Toast + DB-Record + Notification-Flag.

**Ergebnis:**
- HTTP 200 von `receive-event-inquiry`, `inquiryId` zurückgegeben
- Success-Toast erscheint ("innerhalb von 24 Stunden")
- DB-Record angelegt (`v2_events` via Compat-View)
- `notification_sent = true` → E-Mail an Admin ging raus, WhatsApp fire-and-forget getriggert
- Test-Datensatz nachträglich als `is_test=true, archived=true` markiert

**Findings:**

1. **Kein Test-Mode-Flag auf öffentlichem Formular** (P1)
   Der User kann von außen nicht als Test markieren. Jede E2E-Anfrage landet
   erst als echte Anfrage im Inbox+WhatsApp-Alarm und muss danach manuell
   ausgeblendet werden. Fix: `EventContactForm` sollte bei Marker `E2E-TEST-`
   im Firmenname automatisch `isTest: true` an die Function senden — oder
   Query-Param `?test=1` respektieren.

2. **`Firma` ist Pflichtfeld — Privatkunden können nicht anfragen** (P0)
   `z.string().min(2, "Firmenname erforderlich")` in `EventContactForm.tsx:35`.
   Konsequenz: Privatpersonen tippen "Privat", "-", "n/a" → das ist genau die
   Quelle für die "null"-Anzeige-Kaskade aus Bug #1. Fix: Firma optional
   machen ODER B2C/B2B-Toggle einbauen.

3. **Console-Warnungen auf `/events/`** (P2, kosmetisch)
   - `Each child in a list should have a unique "key" prop` in `Navigation.tsx`
   - `React does not recognize the "fetchPriority" prop` auf einem `img`

Keine tatsächlichen Runtime-Errors, keine Netzwerk-Fehler, keine
Fehlermeldungen an den User.
